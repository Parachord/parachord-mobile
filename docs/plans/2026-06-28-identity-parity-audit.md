# IDENTITY-PARITY AUDIT — Legacy → N-way Sync Migration (parachord-mobile#289 / parachord#911)

## 1. Verdict

**NOT SAFE to arm `new`-mode real writes on a mixed legacy↔new fleet yet** — five confirmed identity divergences cause false-removes, cross-client wrong-song dups, or playlist churn on shared remotes; real writes must stay gated (`isNwayPropagateEnabled = OFF`) until D-Legacy-1/2 (hydration) and the norm-collapse cluster (D3/D6) are fixed on both platforms.

---

## 2. Confirmed divergences (isReal=true), ranked by corrected severity

The single most dangerous fact tying these together: **once any client mints an external ID, both engines reconcile by pure provider-ID string equality** — so a wrong ID written by one client is silently treated as "the same track" by every other client on pull. Identity errors are sticky and cross-platform.

### HIGH — D-Legacy-1: Hydration match gate differs (substring + 0.60 floor vs exact-or-items[0] with no floor)
*(verified verdict "D1", legacy-mobile vs legacy-desktop — confidence high)*

- **What diverges:** When minting a provider ID for an ID-less local track from catalog search, **mobile** accepts `items.firstOrNull{playable}` only if `scoreConfidence ≥ 0.60` (bidirectional substring on both title AND artist), else returns null and **drops the track**. **Desktop** accepts the first exact case-insensitive title+artist pair, else falls back to **`|| items[0]` with no confidence floor and no rejection**.
- **Concrete example:** Local "Intro" by The xx; search `items[0]` = "Intro" by Alt-J (different artist). Mobile: artist `thexx` vs `altj` share no substring → 0.50 < 0.60 → **drops**. Desktop: exact pair fails → `|| items[0]` → **writes Alt-J's URI** as The xx's "Intro".
- **Shared-remote corruption:** Desktop writes the wrong URI via `replacePlaylistTracks`; a mobile client pulls, sees the Alt-J URI as a present track, and **keeps it** (URI equality = same track). The fleet converges on a wrong-song entry neither client would independently choose. When the two alternate pushes, the track flaps add/remove.
- **Fix (desktop):** Remove the unguarded `|| items[0]` fallback in `sync-providers/{spotify,applemusic}.js#resolveTracks`; gate the candidate through the shared 0.60 confidence floor so a non-match is dropped rather than mis-minted. This is the root cause of D-Legacy-1, D3-amplification, and D7. **Both platforms should share one hydration-gate contract.**

### HIGH/MEDIUM — D-Legacy-2: Apple Music ISRC-exact tier (mobile) absent on desktop
*(verified verdict "D2", legacy-mobile vs legacy-desktop — confidence high, severity medium)*

- **What diverges:** Mobile `AppleMusicSyncProvider.searchForTrackId` tries an **exact catalog-by-ISRC lookup first** (`getCatalogSongIdByIsrc(storefront, isrc)`) and returns immediately on hit. Desktop legacy `applemusic.js` has **no ISRC tier at all** and hardcodes the **US storefront** (`/catalog/us/`), while mobile uses the **user's actual storefront**. So the two mint different catalog IDs for the same recording — guaranteed for any non-US user even on an exact title+artist match.
- **Concrete example:** ID-less "Redbone" / Childish Gambino, `trackIsrc=USQX91601658`. Mobile (storefront `gb`) → ISRC-exact catalog id `1452906710`. Desktop (us, text search) → a different ranked pressing id.
- **Shared-remote corruption:** AM reconcile is catalog-id **set membership**, and `replacePlaylistTracks` degrades PUT→POST-append on 401 (removals are no-ops). Each client adds its own id and never removes the other's → the playlist **accumulates a permanent duplicate** of the song.
- **Fix (desktop):** Add an ISRC-exact catalog tier to desktop AM hydration AND make it storefront-aware (stop hardcoding `us`). Mobile is already correct.

### MEDIUM — D-Nway-vs-Legacy-3: Cross-provider mirror unification (N-way) vs provider-siloed IDs (legacy)
*(verified verdict "D2-cross-provider-same-isrc-mirror" — confidence high, severity medium; gated OFF today)*

- **What diverges:** N-way `unifyTrackKeys` unions a Spotify-sourced copy and an Apple-sourced copy of one recording via shared ISRC/MBID/norm → **one logical track**. Legacy `extractExternalTrackIds` is single-field-per-provider (spotify→URI, applemusic→numeric id) with **no cross-id-space comparison** → two unrelated entries. (Bites only when the song exists as two separate `PlaylistTrack` rows, one per provider id — the cross-provider-import / partial-hydration shape.)
- **Concrete example:** "Bad Guy" / Billie Eilish, `isrc=USUM71900764`; Row S has only the Spotify URI, Row A only the AM catalog id. N-way → one `isrc-USUM71900764` key; legacy → two siloed entries.
- **Shared-remote corruption:** A user removes the track on the N-way client → N-way drops it from both mirrors. The legacy client still holds it, re-hydrates, and **resurrects it per-provider** (false-remove undo). Conversely a legacy single-mirror add gets cross-mirrored by N-way into a dup. The engines fight over one logical track legacy sees as two.
- **Fix (both):** This is intrinsic to the legacy model and cannot be retrofitted into legacy cheaply. Mitigation is the migration gate itself — **do not run N-way real writes on a playlist a legacy client also pushes**. Long-term: legacy clients must upgrade off provider-siloed identity before N-way arms on shared mirrors.

### MEDIUM — D-Nway-vs-Legacy-4: norm tier collapses two genuinely-distinct recordings
*(verified verdict "D3-norm-collapse-two-distinct-recordings" — confidence high, severity medium; gated OFF today)*

- **What diverges:** N-way's `norm` tier (`<artist>|<title>`) is **always computed** and unions via share-ANY-tier, with the **norm-bridge guard explicitly NOT implemented** (`NwayKeyUnify.kt:81-85`). So two recordings with **different ISRCs and MBIDs** but identical normalized artist|title collapse to one key. Legacy keeps them distinct (different URIs).
- **Concrete example:** Two "Intro" by The xx — album cut (`isrc=GBBKS0700116`) and single edit (`isrc=GBBKS1300999`), distinct URIs. N-way → both rewrite to `isrc-GBBKS0700116` (one track). Legacy → two tracks.
- **Shared-remote corruption:** N-way's merge believes the playlist has one "Intro" and **materialize false-removes** one URI; legacy re-adds it next cycle → flapping.
- **Fix (both):** Implement the **norm-bridge guard** (parachord#911 / docs/plans/2026-06-21-nway-key-consistency-design.md) in BOTH engines before arming: norm may bridge only when stronger tiers are absent or agree; never bridge across confident-but-disagreeing ISRCs/MBIDs. Must land byte-identically on desktop + mobile with shared fixtures.

### MEDIUM — D-Nway-vs-Legacy-5: Remaster-suffix strip (N-way) collapses original + remaster
*(verified verdict "D6-remaster-suffix-strip-asymmetry" — confidence high, severity medium; gated OFF today)*

- **What diverges:** This is D4 specialized to the remaster rule. N-way `normalizeTitleForKey` strips a trailing remaster annotation, so "Wish You Were Here" and "Wish You Were Here (2011 Remastered)" — **different ISRCs, different URIs** — collapse to one norm and (via the missing guard) to one ISRC key. Legacy keeps both URIs.
- **Concrete example:** Pink Floyd original (`isrc=GBN9Y1100123`) + 2011 remaster (`isrc=GBN9Y1100777`). N-way → one `isrc-GBN9Y1100123`. Legacy → two distinct tracks.
- **Shared-remote corruption:** Same false-remove + re-add flap as D4; **especially likely on classic-rock playlists that intentionally pair original + remaster.**
- **Fix:** Same as D4 — the norm-bridge guard fixes both (the guard refuses to bridge across the two distinct confident ISRCs).

### MEDIUM (narrow) — D-Legacy-6: Mobile vs desktop legacy hydration mint different/absent IDs
*(verified verdict "D7-mobile-vs-desktop-legacy-hydration-divergence" — confidence medium, severity medium)*

- **What diverges:** Same root as D-Legacy-1 but framed as two *legacy* clients disagreeing: for an ID-less track whose top catalog hit is a wrong-artist same-title (cover/karaoke/tribute), **mobile drops** (sub-0.60) while **desktop mints `items[0]`**. Playlist contents then depend on which legacy client pushed last; N-way later amplifies the desktop wrong-URI across mirrors.
- **Fix:** Subsumed by D-Legacy-1's fix (remove desktop's ungated `|| items[0]`, share the 0.60 gate). No separate work.

### MEDIUM (mostly benign direction) — D-Nway-vs-Legacy-7: Spotify relink — same ISRC, new track id
*(verified verdict "D1-isrc-relink-same-isrc-new-id" — confidence high, severity medium; gated OFF + norm-bridged)*

- **What diverges:** N-way unifies a relinked track (new Spotify URI, unchanged ISRC) via the ISRC or norm tier → same. Legacy keys on the URI → different → triggers a redundant replace.
- **Reality check:** Overwhelmingly the **benign** direction — N-way's "same" answer is correct and causes no data loss, because the `norm` tier (`fleetwood mac|dreams`, stable across a relink) bridges even when the fetched ISRC is momentarily absent. The harmful false-remove sub-case requires a baseline keyed `isrc-` AND a fetch lacking ISRC AND a *differing* norm — which a relink doesn't produce. Listed for completeness; **no dedicated fix needed** beyond the norm-bridge guard.

## 3. Confirmed parity (do not re-audit)

- **N-way tier extraction is byte-identical, mobile ↔ desktop** for ISRC (trim+UPPER, regex `^[A-Z]{2}[A-Z0-9]{3}[0-9]{7}$`), MBID (trim+lower, non-empty, no UUID validation — both deliberately dropped the strict regex), and norm artist/title (lower+trim, same anchored remaster regex, `<artist>|<title>` with literal pipe). Empirically diffed across 9–16 inputs; identical on every case. *(verdict D1 / D2 of the Nway pair — both isReal=false)*
- **The trailing `.trim()` on mobile after the remaster strip is a verified no-op** (the `\s*$`-anchored regex already consumes trailing whitespace). Not a divergence.
- **The desktop secondary `mbid` fallback (`recordingMbid || mbid`)** is unreachable for `PlaylistTrack` inputs (no `trackMbid` field exists; LB maps its mbid into `trackRecordingMbid`) and is masked by the norm tier even hypothetically. Latent contract drift, not active mis-key.
- **Union-find determinism is identical** (root = smaller index; representative = strongest present tier, lexicographically-min value; UTF-16 code-unit ordering on both).
- **N-way keeps the identity `norm` DISTINCT from the confidence-scoring `normalizeStr`** on both platforms (the merge norm keeps spaces/accents; the confidence norm strips all non-alphanumerics + NFKD). No leakage either direction.
- **PlaylistMerge** (union-add, delete-wins, LWW order, dedup first-occurrence) is byte-identical and fixture-pinned.
- **Legacy reconcile is pure external-ID equality on both platforms** (Spotify URI order-aware; AM catalog-id set membership; LB lowercased recording-MBID order-aware); the LB skip-unchanged short-circuit is LB-only on both.
- **ListenBrainz collaborator-merge is NOT a divergence** *(verdict D6 of legacy pair — isReal=false)*: desktop's collaborator-merge is gated behind `mergeWithRemote` which **defaults false and is set by no caller** (dead code). Both engines wholesale-replace and drop a collaborator-added track — a **shared latent bug**, not a platform gap.
- **ListenBrainz hydration is NOT the claimed divergence** *(verdict D7 of legacy pair — isReal=false)*: desktop is NOT JSPF-only — `resolveTrackMbid` hits the LB mapper (≥0.7) just like mobile's `getRecordingMbid`. They agree on the common case. A real but **much narrower** disagreement exists only at mobile's last-resort fuzzy MB search (mapper-miss + ISRC-miss); cross-artist substitution is impossible (artist-axis gate).
- **MBID-enrichment upgrade window** *(verdict D5-mbid-enrichment — isReal=false)*: the always-present, baseline-persisted `norm` tier bridges enrichment upgrades; the unconditional one-cycle baseline advance self-heals. Only a rare title-change-concurrent-with-enrichment would split, and only in shadow mode.

## 5. Recommended next actions for #289 (ordered)

1. **Keep `isNwayPropagateEnabled` OFF.** Do not arm real writes on shared remotes until items 2–4 land on both platforms. (D-Nway-vs-Legacy-3/4/5/7 are all gated off today — this is the load-bearing safety.)
2. **Fix desktop legacy hydration (D-Legacy-1 + D-Legacy-6):** remove the ungated `|| items[0]` fallback in `sync-providers/{spotify,applemusic}.js`; route candidates through the shared 0.60 confidence gate; drop on sub-floor. Highest-severity, fixable independent of N-way, and prevents wrong-song IDs from poisoning shared remotes *right now* under pure legacy operation.
3. **Fix desktop AM hydration (D-Legacy-2):** add an ISRC-exact catalog tier and make hydration storefront-aware (stop hardcoding `/catalog/us/`). Confirm mobile and desktop pick the same catalog id for a given ISRC + storefront.
4. **Implement the norm-bridge guard (D-Nway-vs-Legacy-4 + D-Nway-vs-Legacy-5) in BOTH engines**, byte-identically, with new shared fixtures covering disagreeing-ISRC-same-norm and original-vs-remaster pairs. This is the precondition for arming N-way without false-removing legitimately-distinct recordings.
5. **Build the no-false-drop harness** (already a tracked gate): run sync×2 + a legacy↔new alternation scenario asserting zero false-removes / zero dup-adds on shared Spotify, AM, and LB mirrors, with the D-prefixed example pairs above as explicit cases.
6. **Re-verify all desktop-side claims against `origin/main`** (uncertainty #1) and diff both engines against the shared fixture files (uncertainty #2) before flipping the gate.
7. **Separately track the two shared latent bugs surfaced here** (not migration blockers): the LB collaborator-merge dead-code drop (both engines wholesale-replace and lose collaborator tracks), and the AM append-only no-removal duplicate accumulation.

**Confidence summary:** Divergence *mechanisms* are high-confidence (verified against live mobile source; desktop via consistent dual-model extraction). The *gating* makes the three N-way↔legacy divergences inert today; the two legacy↔legacy hydration divergences (D-Legacy-1/2) are **live now** under pure legacy operation and warrant fixing regardless of the #289 timeline.
